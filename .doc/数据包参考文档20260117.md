# 数据包参考文档

> 更新时间：2026.01.17

## 服务器和客户端数据包

### 📤 客户端 → 服务器（请求）

| 消息类型         | Type   | 格式                          | 说明                                       |
| ---------------- | ------ | ----------------------------- | ------------------------------------------ |
| **创建房间**     | `0x01` | `[type:1][nameLen:1][name:n]` | nameLen 为房间名长度，name 为 UTF-8 字符串 |
| **查询房间列表** | `0x02` | `[type:1]`                    | 无载荷                                     |
| **加入房间**     | `0x03` | `[type:1][roomId:4]`          | roomId 为大端序 uint32                     |
| **开始游戏**     | `0x05` | `[type:1]`                    | 仅房主可发送                               |
| **退出房间**     | `0x06` | `[type:1]`                    | 房主退出并销毁房间                         |
| **离开房间**     | `0x07` | `[type:1]`                    | 房客离开房间                               |

### 📥 服务器 → 客户端（响应）

| 消息类型         | Type   | 格式                                      | 说明                             |
| ---------------- | ------ | ----------------------------------------- | -------------------------------- |
| **房间已创建**   | `0x81` | `[type:1][length:2][roomId:4]`            | roomId 为大端序 uint32           |
| **房间列表**     | `0x82` | `[type:1][length:2][count:1][rooms...]`   | 见下方详细格式                   |
| **加入结果**     | `0x83` | `[type:1][length:2][success:1][roomId:4]` | success=1 成功，roomId 为大端序  |
| **房客加入通知** | `0x84` | `[type:1][length:2][roomId:4]`            | 通知房主有人加入                 |
| **对战开始**     | `0x85` | `[type:1][length:2]`                      | 进入中继模式，后续数据为对战数据 |
| **房间已退出**   | `0x86` | `[type:1][length:2]`                      | 确认退出/离开成功                |
| **房客离开通知** | `0x87` | `[type:1][length:2][roomId:4]`            | 通知房主有人离开                 |
| **错误**         | `0xFF` | `[type:1][length:2][errorCode:1]`         | 错误码                           |



## 客户端转发数据包

所有数据包继承自 `BaseEvent`：

```
[type:1][size:1][payload:0~n]
```

| 字段    | 大小      | 说明                            |
| ------- | --------- | ------------------------------- |
| type    | 1 byte    | 事件类型 (EventType 枚举)       |
| size    | 1 byte    | 数据包总大小（含 type 和 size） |
| payload | 0~n bytes | 载荷数据                        |

## 数据包设计

以下内容为个人想法，仅供参考

```
[                        协议头 : 16byte                        ][负载内容]
[协议版本:1][请求类型:1][请求类型子类型:1][帧长度:4][魔数:1][时间戳:8]
```

- 协议版本：避免数据包协议版本不同导致的解析出错
- 请求类型：参考HTTP协议GET、POST等请求类型
- 请求类型子类型：相当于 Restful 风格中请求的资源
- 帧长度：包括协议头8个字节
- 魔数：作为填充，也可以作为 token 使用
- 时间戳：为异步程序提供按时间顺序同步的支持



## 转发阶段组帧笔记（20260117）

此部分记录了客户端双方经过服务器转发流量时，服务器进行拦截并解析的思路和存储方法

主要以文件存储为主，遵循以下格式：

- 对于相邻两个事件之间，以**换行符（`\n`）**分隔
- 对于一个数据包，相邻数据之间使用 **$$** 和**若干空格**进行分隔。前者是必要的，后者可有可无
- 每个数据包开始存储时先描述 发生的时刻（long类型，使用**+**起头） 和 事件类型（使用**@**符号起头），两者之间若干空格分隔。之后要跟上第二点提到的分隔符，避免解析出错

例如以下示例（仅作参考）

```
+10001 @金币添加    $$ 32 $$ 8 $$ 1 $$ 1 $$ 10 $$ 10
+10003 @植物开火 $$    42 $$ 14 $$ 233 $$ 244 $$ 5 $$ 0 $$ 20 $$ 1
+10010 @认输同步   $$ 31 $$ 2
```

对于 Netty 框架，可以将解析器附加到转发处理器上异步运行，这样既能保证性能，代码入侵度也比较小
